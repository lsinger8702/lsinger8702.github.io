MySQL 索引

Posted on 2021-06-06 Edited on 2021-06-07 In MySQL

概念

  • 索引是存储引擎用于可快速找到记录的一种数据结构
  • 索引是在存储引擎层，不同存储引擎的索引工作方式不同

索引类型

MySQL支持的索引类型有三种

B+ Tree 索引

原理

  • 所有值按顺序存储，每个叶子页到根的距离相同
  • 加快访问数据的速度，不需要进行全表扫描，从索引的根节点开始进行向下层查找
    • 比较节点页的值和要查找的值找合适的指针进入下层节点
    • 直到找到指向被索引的数据的叶子节点
  • 所有记录节点都是按键值的大小顺序存放在同一层的叶子节点上，由各叶子节点指针进行连接

优点

  • 适用于顺序、全键值、键值范围或键前缀查找（只适用根据最左前缀查找）

限制

  • 如果不是按索引的最左列开始查找，则无法使用索引
  • 不能跳过索引中的列
  • 如果查询中有某个列的范围查询，其右边所有列都无法使用索引优化查找

哈希索引

原理

  • 基于哈希表实现，只有精确匹配索引所有列的查询才有效
  • 对每行数据，引擎对所有的索引列计算一个哈希码
  • 哈希码存储在索引中，哈希表中保存指向每行数据的指针
  • 只有 Memory 引擎显示支持哈希索引，也是它的默认索引类型

限制

  • 不保存索引的值，所以不能使用索引的值来避免读取行
  • 并不是按照索引顺序存储，无法用于排序
  • 不支持部分索引列匹配查找
  • 只支持等值比较查询，不支持任何范围查询
  • 哈希冲突很多时，索引维护的代价很高

自适应哈希索引

  • 出现哈希冲突时，必须遍历链表中的所有行指针

  • 必须在where子句中包含常量值来避免哈希冲突

  • innodb有一个特殊的功能叫做自适应哈希索引

    • 当某个索引值被使用的非常频繁时，会在内存中基于B树索引之上再创建哈希索引
    • 使用哈希值而不是键本身进行查找，要在查询的where语句中指定使用哈希函数
    • 根据哈希值做快速的整数比较找到索引条目，然后一一比较返回对应的行

空间数据索引（R-Tree）

#####全文索引

  • 查找的是文本中的关键词
  • 类似于搜索引擎，而不是简单地where条件匹配
  • 可以同时创建全文索引和B-Tree不会有冲突
  • 适用于 MATCH AGAINST操作而不是 WHERE条件操作

  • Innodb

    • 聚集索引
      • 按照每张表的主键构造一颗B+树，叶子节点（数据页）中存放的即为整张表的行记录数据
        • InnoDB存储引擎表是索引组织表，表中数据按照逐渐顺序存放
        • 数据是索引的一部分
        • 每个数据页通过一个双向链表进行链接
        • 只有一个聚集索引
        • 存储在物理上不连续，逻辑上连续

几种数据结构区别

B-Tree v.s. B+ Tree

  • B+树是B树的一个升级版，B+树更充分的利用了节点的空间，让查询速度更加稳定，其速度完全接近于二分法查找
  • B+树的层级更少，B+每个非叶子节点存储的关键字数更多，树的层级更少所以查询数据更快
  • B+树查询速度更稳定，B+所有关键字数据地址都存在叶子节点上，所以每次查找的次数都相同所以查询速度要比B树更稳定
  • B+树天然具备排序功能，B+树所有的叶子节点数据构成了一个有序链表，在查询大小区间的数据时候更方便，数据紧密性很高，缓存的命中率也会比B树高。
  • B+树全节点遍历更快，B+树遍历整棵树只需要遍历所有的叶子节点即可，而不需要像B树一样需要对每一层进行遍历，这有利于数据库做全表扫描
  • B树相对于B+树的优点是，如果经常访问的数据离根节点很近，而B树的非叶子节点本身存有关键字其数据的地址，所以这种数据检索的时候会要比B+树快
